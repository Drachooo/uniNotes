    \chapter{Introduzione}
    La nascita dei sistemi per la gestione di basi di dati ha visto due momenti principali:
    \begin{description}[labelindent=0.5cm]
      \item [anni '60]: \underline{sviluppo di applicazioni} negli \textcolor{azzurrino}{\textbf{ambienti di ricerca scientifica}}
      \item [anni '70]: \underline{sviluppo di applicazioni} informatiche in \textcolor{azzurrino}{\textbf{ambito gestionale}}  
    \end{description}

    Si trattava di semplici dispositivi in cui gli algoritmi di elaborazione erano semplici e grandi quantità di dati erano condivisi da più applicazioni.\\[0.2ex]
    Queste caratteristiche erano specifiche per l'ambiente in cui erano state introdotte ovvero il \textcolor{azzurrino}{\textbf{sistema informativo}}.

    \definizione{sistema informativo}{
      Un \textcolor{azzurrino}{\textbf{sistema informativo}} è l'\hl{insieme delle attività umane} e dei \hl{dispositivi di memorizzazione ed elaborazione} che \underline{organizza e gestisce l'informazione di interesse} per un'organizzazione di dimensioni qualsiasi.
    }

    Il sistema informativo è costituito da \textbf{dati} e \textbf{informazioni}:
    \begin{description}
      \centering
      \item [\textcolor{azzurrino}{dato}]
      
      \item elemento di conoscenza di base costituito da simboli che devono essere elaborati
      
      \item $\downarrow$
      
      \item [\textcolor{azzurrino}{informazione}]
      
      \item interpretazione dei dati che permette di ottenere conoscenza più o meno esatta di fatti e situazioni 
    \end{description}

    \medskip

    Lo studio del sistema informativo avviene attraverso i \textbf{diagrammi di flusso}, i quali permettono di svolgere diverse operazioni tra cui:
    \begin{itemize}
      \item definizione di archivi di dati e delle sorgenti di dati
      \smallskip
      \item definiazione degli utenti
      \smallskip
      \item definizione di procedure e processi
      \smallskip
      \item definizione dei flussi di dati
    \end{itemize}

    \section{Nascita dei DBMS}
    Negli anni '70 i \hl{programmi comunicavano direttamente con i dati contenuti dei file system}.

    Si trattava di:
    \begin{itemize}[label=$\triangleright$]
      \item \textcolor{azzurrino}{\textbf{operazione scomoda}}: l'accesso ai dati su file era scarso
      \smallskip
      \item \textcolor{azzurrino}{\textbf{rindonza nei dati}}: duplicazione dello stato dato su più file
      \smallskip
      \item \textcolor{azzurrino}{\textbf{inconsistenza}}: dovuta ad aggiornamenti parziali
      \smallskip
      \item \textcolor{azzurrino}{\textbf{progettazione dei dati era replicata per ogni programma}}
    \end{itemize}

    Negli anni '80 la \hl{soluzione che venne proposta fu quella di inserire un \textbf{DBMS}} (Data Base Management System) tra file system e applicazioni.

    \begin{figure}[H]
      \centering
      \begin{subfigure}[b]{0.45\textwidth}
        \centering
        \includegraphics[width=\textwidth]{images/fileSystem.png}
        \caption{file system}
      \end{subfigure}\hfill
      \begin{subfigure}[b]{0.45\textwidth}
        \centering
        \includegraphics[width=\textwidth]{images/DBMS.png}
        \caption{file system + base di dati}
      \end{subfigure}
      \caption{prima e dopo l'innovazione}
    \end{figure}

    \definizione{base di dati}{
      Una \textcolor{azzurrino}{\textbf{base di dati}} è una \hl{collezione di dati} dati utilizzati per rappresentare con la tecnologia informatica le informazioni di interesse per un sistema informativo.
    }

    \definizione{DBMS}{
      Un \textcolor{azzurrino}{\textbf{DBMS}} è un \hl{sistema che gestisce} su memoria secondaria \hl{collezioni di dati} grandi, condivisi e persistenti, assicurandone l'\textbf{affidabiltà}, la \textbf{privatezza} e l'\textbf{accesso efficiente}.
    }

    Questo nuovo approcio ha portato a numerosi vantaggi:
    \begin{itemize}[label=$\Rightarrow$]
      \item \textbf{maggiore astrazione} e \textbf{potenza espressiva} per descrivere le proprietà del dato.\\[0.2ex]
      \hl{All'interno di un DBMS, i dati vengono interpretati come \textbf{oggetti}} (istanze di classi o righr di tabelle).\\[0.2ex]
      Prima erano interpretati come blocchi o pagine di memoria secondaria (sequenze di byte).

      \medskip

      \item \textbf{operazioni di asccesso di dati più complesse} basate su un linguaggio di interrogazione (\textit{select from where}), anzichè attraverso operazioni di \textit{read/write}
      
      \medskip

      \item \textbf{migliore interazione uomo-informazione}:
        \begin{itemize}[label=$\star$]
          \item \textcolor{azzurrino}{\textbf{linguaggio per la definizione dei dati}} $\rightarrow$ \textcolor{azzurrino}{\textbf{D}}ata \textcolor{azzurrino}{\textbf{D}}efinition \textcolor{azzurrino}{\textbf{L}}enguage
          
          \smallskip

          \item \textcolor{azzurrino}{\textbf{linguaggio di manipolazione}}: \underline{popola} la base di dati e \underline{modifica} il suo contenuto con aggiunte, cancellazioni e variazioni sui dati (es. SQL)
        \end{itemize}
    \end{itemize}

    \definizione{modello di dati}{
      Un \textcolor{azzurrino}{\textbf{modello di dati}} è l'\hl{insieme di costrutti forniti dal DBMS} per descrivere la struttura e le proprietà dell'informazione contenuta in una base di dati.

      \smallskip

      I costrutti permettono di definire le strutture dati che conterranno le informazioni e specificare le proprietà che dovranno soddisfare le istanze.
    }

    Nel tempo si sono susseguiti diversi modelli di dati:
    \begin{itemize}[label=$\Rightarrow$]
      \item \textbf{nel passato}: gerarchico, reticolare
      \smallskip
      \item \textbf{attualmente}: relazionale, a oggetti, object-relational (SQL:1999), document-oriented (Json), NoSQL
    \end{itemize}

    \definizione{schema di una base di dati}{
      Lo \textcolor{azzurrino}{\textbf{schema di una base di dati}} (invariante nel tempo) è la \hl{descrizione della struttura e delle proprietà} di una specifica base di dati fatta utilizzando i costrutti del modello dei dati.
    }

    \definizione{istanza di una base di dati}{
      L'\textcolor{azzurrino}{\textbf{istanza di una base di dati}} (variante nel tempo) è costituita dai \hl{valori che in un certo istante popolano le strutture dati} della base di dati.
    }

      \centering
      \includegraphics[width=0.6\textwidth]{images/modelloSchemaIstanza.png}
      \caption{distinzione tra modello, schema e istanza}
    \end{figure}

    \section{Architettura dei DBMS}
    Lo schema di una base di dati è diviso in tre livelli:
    \begin{description}
      \centering
      \item[\textcolor{azzurrino}{livello interno}]
      \item rappresenta la base di dati per mezzo delle strutture fisiche di memorizzazione (file dati, file indice, ecc.)
      
      \smallskip

      \item $\downarrow$
      
      \smallskip

      \item[\textcolor{azzurrino}{livello logico}]
      \item rappresenta la struttura e le proprietà della base di dati definita attraverso i costrutti del modello dei dati del DBMS
      
      \smallskip

      \item $\downarrow$
      
      \smallskip

      \item[\textcolor{azzurrino}{livello esterno}]
      \item descrive una porzione dello schema logico di interesse per uno specifico utente o applicazione
    \end{description}

    \medskip

    La caratteristica fondamentale di queste strutture è l'\textbf{indipendenza}:
    \begin{itemize}[label=$\star$]
      \item \textcolor{azzurrino}{\textbf{fisica}}: \hl{schema logico completamente indipendente dallo schema interno}.
      
      Ne consegue che le variazioni delle strutture fisiche non impattano sullo schema logico e, quindi, sulle applicazioni

      \smallskip

      \item \textcolor{azzurrino}{\textbf{logica}}: \hl{schemi esterni indipendenti dallo schema logico}.
      
      Ne consegue che le variazioni dello schema logico (senza rimuovere nessun dato) non impattano sugli schem esterni e, quindi, sulle applicazoni
    \end{itemize}



    \chapter{Progettazione di una base di dati}
    La \textbf{progettazione di una base di dati} costituisce solo una delle componenti del processo di sviluppo di un sistema informativo e va, quindi, inquadrata in un contesto più ampio: \hl{\textbf{ciclo di vita} dei sistemi informativi}.

    Il ciclo di vita di un sistema informativo comprende:
    \begin{description}
      \centering
      \item [\textcolor{azzurrino}{studio della fattibilità}] 
      \item definisce i costi delle varie alternative possibili e stabilisce le proprietà di realizzazione delle varie componenti del sistema
      \smallskip
      \item $\downarrow$
      \smallskip
      \item [\textcolor{azzurrino}{raccolta  e analisi dei requisiti}]
      \item individua e studia le proprietà e le funzionalità del sistema, producendo una descrizione completa, ma informale dei dati coinvolti e delle operazioni su di essi
      \smallskip
      \item $\downarrow$
      \smallskip
      \item [\textcolor{azzurrino}{progettazione}]
      \item $\rightarrow$ \textit{progettazione dei dati}: descrive la struttura e l'organizzazione dei dati
      
      $\rightarrow$ \textit{progettazione delle applicazioni}: descrive in modo formale le caratteristiche dei programmi applicativi
      
      Le due attività attività sono complementari e possono procedere in parallelo o a cascata
      \smallskip
      \item $\downarrow$
      \smallskip
      \item [\textcolor{azzurrino}{implementazione}]
      \item realizza il sistema informativo secondo la struttura e le caretteristiche definite nella fase di progettazione.
      
      Viene costruita e popolata la base di dati e viene prodotto il codice dei programmi
      \smallskip
      \item $\downarrow$
      \smallskip
      \item [\textcolor{azzurrino}{validazione e collaudo}]
      \item verifica il corretto funzionamento e la qualità del sistema informativo.
      
      La sperimentazione deve prevedere, per quanto possibile, tutte le condizioni operative
      \smallskip
      \item $\downarrow$
      \smallskip
      \item [\textcolor{azzurrino}{funzionamento}]
      \item entra in funzione il sistema informativo ed esegue i compiti per i quali era stato originariamente progettato.
      
      Se non si verificano malfunzionamenti o revisioni delle funzionalità del sistema, questa attività richiede solo operazioni di gestione e manutenzione
    \end{description}

    \section{Progettazione dei dati}
    Una metodologia di progettazione è costituita da:
    \begin{itemize}[label=$\triangleright$]
      \item \textbf{decomposizione}: dividere in passi (indipendenti) le attività di progetto
      \smallskip
      \item \textbf{strategie}: individuare un insieme di strategie e criteri di scelta da seguire
      \smallskip
      \item \textbf{modelli di riferimento}: utilizzare modelli di dati e tecniche di progettazione consolidate
    \end{itemize}

    Una buona metodologia deve essere:
    \begin{itemize}
      \item \textcolor{azzurrino}{\textbf{generale}}: rispetto al problema studiato e agli strumenti utilizzati
      \smallskip
      \item \textcolor{azzurrino}{\textbf{prodotto di qualità}}: in termini di \underline{correttezza}, \underline{completezza} e \underline{efficienza} delle risorse impiegate
      \smallskip
      \item \textcolor{azzurrino}{\textbf{facile da usare}}: nelle strategie e nei modelli di riferimento
    \end{itemize}

    \section{Requisiti}
    Nell'ambito delle basi di dati si è consolidata negli anni una metodologia di progettazione che ha dato prova di soddisfare pienamente le proprietà descritte.\\[0.2ex]
    Tale metodologia è articolata in tre fasi principali da efettuare a cascata.

    \subsection{Progettazione concettuale}
    La \textcolor{azzurrino}{\textbf{progettazione concettuale}} è la \hl{prima fase della progettazione dei dati}.\\[0.2ex]
    Lo scopo è quello di \underline{produrre una descrizione formale dei dati} (schema concettuale).\\[0.2ex] 
    Lo schema deve essere indipendente dall’implementazione.

    Non è solo un progetto intermedio, ma costituisce anche una porzione del risultato finale perchè rappresenta una descrizione di alto livello del contenuto della base di dati, comprensibile anche per utenti poco esperti.

    \subsection{Progettazione logica}
    La \textcolor{azzurrino}{\textbf{progettazione logica}} è la \hl{seconda fase della progettazione dei dati}.\\[0.2ex]
    Lo scopo è quello di \underline{tradurre lo schema concettuale in uno schema logico} in modo da poterlo utilizzare su un sistema specifico.
    
    Lo schema logico infatti è dipendente dalle tecnologie utilizzate e bisogna tenere anche in considerazione le operazioni più frequenti che le applicazioni effettueranno sulla base di dati.

    \subsection{Progettazione fisica}
    La \textcolor{azzurrino}{\textbf{progettazione fisica}} è la \hl{terza fase della progettazione dei dati}.\\[0.2ex]
    Lo scopo è quello di \underline{ottimizzare l’accesso ai dati} completando lo schema logico con i parametri relativi alla memorizzazione fisica dei dati e con gli opportuni metodi d’accesso (\textit{indici}).

    \section{Stategie di progettazione}
    Lo sviluppo do uno schema concettuale può essere visto come un processo di ingegnerizzazione ed è, quindi, possibile applicare anche a tale processo le strategie classiche:
    \begin{itemize}[label=$\diamond$]
      \item \textbf{top-down} $\downarrow$ : consiste nel \hl{considerare le specifiche globalmente} e produrre uno schema iniziale completo ma con pochi concetti molto astratti.
      
      Si va poi a raffinare i concetti astratti fino ad arrivare allo schema concettuale completo in ogni dettaglio

      \smallskip

      \item \textbf{bottom-up} $\uparrow$ : consiste nel \hl{decomporre le specifiche iniziali in parti elementari} e produrre uno schema iniziale dettagliato
      
      \smallskip
      
      \item \textbf{inside-out}: consiste nell’\hl{individuare nelle specifiche alcuni concetti importanti} (\textit{concetti guida}) e partendo da quelli generare gli schemi per i relativi concetti.\\[0.2ex]
    Si procede fondendo gli schemi precedenti per generare lo schema finale
    \end{itemize}




    \chapter{Progettazione concettuale}
    Il \textbf{modello entità-relazione} (E-R) è un \hl{modello formale e non ambiguo}, utilizzato per la progettazione concettuale di una base di dati.\\[0.2ex]
    Fornisce una serie di strutture (\textit{costrutti}), con sintassi grafica, per specificare la struttura e le proprietà dei dati da rappresentare indipendentemente dalla tecnologia.

    \section{Entità}
    \definizione{entità}{
      Un'\textcolor{azzurrino}{\textbf{entità E}} rappresenta una \hl{classe di oggetti} con:
    \begin{itemize}[label=$\triangleright$]
      \item \textbf{proprietà comuni}
      \smallskip
      \item \textbf{esistenza autonoma}: rispetto ad altre classi di oggetti
      \smallskip
      \item \textbf{indipendenza univoca}: esiste una chiara corrispondenza tra gli oggetti istanze di entità e concetti istanziati nel sistema informativo
    \end{itemize}
    }

    Un entità si rappresenta con un rettangolo che contiene il nome dell’entità:

    \begin{figure}[H]
      \centering
      \includegraphics[width=0.5\textwidth, keepaspectratio]{images/entità.jpeg}
      \caption{entità}
    \end{figure}

    Un’\hl{istanza dell’entità E è un oggetto appartenente alla classe rappresentata da E}.\\[0.2ex]
    Si indica con $\boldsymbol{\mathrm{I(E)}}$ l’insieme delle istanze di E che esistono nella base di dati in un certo istante e alla creazione della base di dati è vuota ($\mathrm{I(E)} = \emptyset$).

    \section{Relazione}
    \definizione{relazione}{
      Una \textcolor{azzurrino}{\textbf{relazione}} rappresenta un \hl{legame logico tra due o più entità}.
      
      Può esserci anche una relazione all’entità stessa (relazione ricorsiva).
    }

    Una relazione si rappresenta nello schema con un rombo collegato con delle linee alle entità coinvolte nella relazione e a fianco il nome della relazione:

    \begin{figure}[H]
      \centering
      \includegraphics[width=0.5\textwidth, keepaspectratio]{images/relazioniEntità.jpeg}
      \caption{relazione tra entità}
    \end{figure}

    Data una relazione R tra n entità $\mathrm{E_1}, \dots , \mathrm{E_n}$ un’\hl{istanza della relazione R è una ennupla di istanze di entità}:
    \begin{equation*}
      (e_1, \dots, e_\mathrm{n}) \ \text{dove } e_\mathrm{i} \in \mathrm{I(E_i)} \ \text{per } 1 \le \mathrm{i} \le \mathrm{n}
    \end{equation*}

    La popolazione di R rappresenta l’insieme delle coppie di istanze delle entità E e F che sono in relazione in un certo istante:
    \begin{equation*}
      \mathrm{I(R)} = \{(e_\mathrm{i}, f_\mathrm{j}) \ | \ \mathrm{I(E)}, f_\mathrm{j} \in \mathrm{I(F)}\}
    \end{equation*}

    Data una relazione R tra n entità $\{\mathrm{E_1}, \mathrm{E_2}, \dots , \mathrm{E_n}\}$ vale sempre la seguente proprietà sull’insieme delle istanze $\mathrm{I(R)}$:
    \begin{equation*}
      \mathrm{I(R)} \subseteq \mathrm{I(E_1)} \times \mathrm{I(E_2)} \times \dots \times \mathrm{I(E_n)}
    \end{equation*}

    La conseguenza di questa proprietà è che non è possibilie rappresentare la stessa ennupla più volte.

    \definizione{relazione ricorsiva}{
      Una \textcolor{azzurrino}{\textbf{relazione ricorsiva}} è una \hl{relazione binaria sulla stessa entità}.

      \begin{figure}[H]
        \centering
        \includegraphics[width=0.3\textwidth]{images/relazioniRicorsive.jpeg}
        \caption{relazione ricorsiva}
      \end{figure}
    }

    \definizione{relazione ternaria}{
      Una \textcolor{azzurrino}{\textbf{relazione ternaria}} è una \hl{relazione che coinvolge tre entità} e si usa quando:
      \begin{itemize}[label=$\diamond$]
        \item  un concetto, per esistere, nel sistema informativo richiede sempre la presenza di tre istanze di entità (una per ogni entità coinvolta nella relazione)
        \smallskip
        \item non esistono eccezioni al punto precedente
        \smallskip
        \item non esistono situazioni nelle quali la terna di entità debba essere rappresentata più volte
      \end{itemize}

      \begin{figure}[H]
        \centering
        \includegraphics[width=0.4\textwidth]{images/relazioneTernaria.jpeg}
        \caption{relazione tra tre entità}
      \end{figure}

      Un uso errato avviene quando per rappresentare lo stato del sistema informativo sarebbe necessario che un’istanza della relazione contenga solo una coppia e non una terna di istanze di entità (cioè quando manca un attore).
      
      Quindi, in questo caso si può sostituire la relazione ternaria con tre relazioni binarie indipendenti.
    }

    \esempio{uso corretto della relazione ternaria}{
      Vogliamo rappresentare il fatto che un profotto venga fornito ad un dipartimento da un fornitore.

      Ogni fornitore può scegliere quali prodotti ordinare dai fornitori selezionando un sottoinsieme dei prodotti forniti dal fornitore.

      Ogni fornitore vende a più dipartimenti prodotti diversi.

      Ogni prodotto viene venduto da più fornitori.

      \begin{figure}[H]
        \centering
        \includegraphics[width=0.4\textwidth]{images/relazione_ternaria.png}
      \end{figure}

      dove le istanze hanno le seguenti caratteristiche:
      \begin{itemize}[label=$\diamond$]
        \item $\mathrm{I(FORNITORE)} = \{1, 2, 3\}$
        \smallskip
        \item $\mathrm{I(PRODOTTO) = \{p_1, p_2, p_3\}}$
        \smallskip
        \item $\mathrm{I(DIPARTIMENTO) = \{d_1, d_2\}}$
        \smallskip
        \item $\mathrm{I(FORNITURA) = \{(f_1, d_1, p_1), (f_1, d_2, p_3), (f_3, d_1, p_3), (f_3, d_2, p_2)\}}$
      \end{itemize}
    }

    \section{Attributo}
    \definizione{attributo}{
      Un \textcolor{azzurrino}{\textbf{attributo}} \hl{rappresenta una proprietà elementare} di un’entità o di una relazione.
      
      Ogni attributo di un’entità o di una relazione associa ad ogni istanza un solo valore appartenente ad un dominio di valori ammissibili.
    }

    Può essere visto come una funzione che ha come dominio le istanze dell’entità (o relazione) e come codominio l’insieme dei valori ammissibili:
    \begin{equation*}
      f_\mathrm{A} : \mathrm{I(E)} \rightarrow \mathrm{D}
    \end{equation*}

    dove:
    \begin{description}[labelindent=0.5cm]
      \item [A:] attributo dell'entità E
      \smallskip
      \item [$\boldsymbol{\mathrm{I(E)}}$:] insieme delle istanze di E
      \smallskip
      \item [D:] insieme dei valori ammissibili

    \section{Cardinalità}
    \definizione{cardinalità}{
      Data una relazione R i \textcolor{azzurrino}{\textbf{vincoli di cardinalità}} vengono specificati per ogni entità $\mathrm{E_i}$ coinvolta nella relazione R e \hl{specificano il numero minimo e massimo di istanze di R} a cui un istanza di $\mathrm{E_j}$ deve o può partecipare.
    }

    La sintassi grafica per rappresentare i vincoli di cardinalità è la seguente:
    \begin{figure}[H]
      \centering
      \includegraphics[width=0.5\textwidth]{images/cardinalità-relazioni.png}
    \end{figure}

    \vario{Valori possibili per $\boldsymbol{\mathrm{MIN_i}}$}{
      I possibili valori che il minimo $\mathrm{MIN_i}$ può assumere sono:
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{0}: \underline{partecipazione} alla relazione R delle istanze di $\mathrm{E_i}$ è \underline{opzionale}
        \smallskip
        \item \textbf{1}: \underline{partecipazione} alla relazione R delle istanze di $\mathrm{E_i}$ è \underline{obbligatoria}
        \smallskip
        \item \textbf{num > 1}: per ogni istanza di $\mathrm{E_i}$ devono essere presenti \underline{almeno num occorrenze} della relazione R che la coinvolgono
      \end{itemize}
    }

    \vario{Valori possibili per $\boldsymbol{\mathrm{MAX_i}}$}{
      I possibili valori che il minimo $\mathrm{MAX_i}$ può assumere sono:
      \begin{itemize}
        \item \textbf{1}: un’istanza di $\mathrm{E_i}$ può al massimo partecipare a \underline{una sola occorrenza} della relazione R (se R è binaria questo indica che R è una funzione)
        \smallskip
        \item \textbf{N}: un’istanza di $\mathrm{E_i}$ può partecipare a \underline{più occorrenze} della relazione R senza limite massimo
        \smallskip
        \item \textbf{num > 1}: per ogni istanza di $\mathrm{E_i}$ possono essere presenti \underline{al massimo num occorrenze} della relazione R che la coinvolgono
      \end{itemize}
    }

    \section{Generalizzazione di entità}
    \definizione{generalizzazione}{
      La \textcolor{azzurrino}{\textbf{generalizzazione}} è un \hl{legame logico} (simile ad un’ereditarietà tra classi) tra un’entità padre E e n (n > 0) entità figlie ($\mathrm{E_1}, \mathrm{E_2}, \dots , \mathrm{E_n}$) dove E rappresenta una classe di oggetti più generale rispetto alle classi di oggetti rappresentate dalle entità figlie.
    }

    La sintassi grafica per rappresentare una generalizzazione è la seguente:
    \begin{figure}[H]
      \centering
      \includegraphics[width=0.5\textwidth]{images/generalizzazione.png}
    \end{figure}

    \vario{Proprietà delle istanze generalizzate}{
      Le proprietà delle istanze generalizzate sono:
      \begin{itemize}[label=$\triangleright$]
        \item ogni istanza di un’entità figlia $\mathrm{E_i}$ è anche istanza dell’entità padre E
        \smallskip
        \item ogni proprietà (attributi, identificatori e relazioni) dell’entità padre E è anche proprietà di ogni entità figlia $\mathrm{E_i}$
      \end{itemize}
    }

    \vario{Classificazione delle generalizzazioni}{
      Una generalizzazione può essere classificata secondo due criteri:
      \begin{itemize}[label=$\diamond$]
        \item \textcolor{azzurrino}{\textbf{totale}}: se ogni istanza dell’entità padre E è anche istanza di almeno una delle entità figlie $\mathrm{E_i}$, altrimenti si dice \textbf{parziale}
        \smallskip
        \item \textcolor{azzurrino}{\textbf{esclusiva}}: se ogni istanza dell’entità padre E è anche istanza di al massimo una delle entità figlie $\mathrm{E_i}$, altrimenti si dice \textbf{sovrapposta}
      \end{itemize}

      \medskip

      Per indicare la classificazione di una generalizzazione si mettono le seguenti notazioni accanto alla freccia della generalizzazione:
      \begin{itemize}[label=$\star$]
        \item \textbf{(p,s)}
        \smallskip
        \item \textbf{(p,e)}: non si possono avere istanze comuni tra le entità figlie
        \smallskip
        \item \textbf{(t,s)}: non si possono avere istanze dell’entità padre che non siano istanze di almeno una delle entità figlie
        \smallskip
        \item \textbf{(t,e)}: ogni istanza dell’entità padre è istanza di esattamente una delle entità figlie
      \end{itemize}
    }


    

    \chapter{Progettazione logica}
    Nel corso degli anni sono stati sviluppati diversi modelli:
    \begin{description}
      \centering
      \item \textbf{modello gerarchico/reticolare} (anni '60-'70)
      \item $\downarrow$
      \item \textcolor{azzurrino}{\textbf{modello relazionale}} (anni '80-oggi)
      \item $\downarrow$
      \item \textbf{modello ad oggetti/relazionale} o \textbf{object-relstional} (anni 2000-oggi)
      \item $\downarrow$
      \item \textbf{modello a grago - RDF} (anni 2005-oggi) $\Rightarrow$ \textit{Neo4j}
      \item $\downarrow$
      \item \textbf{modello noSQL} (anni 2010-oggi) $\Rightarrow$ \textit{MongoDB}, ecc.
    \end{description}
    
    \noindent\rule{\linewidth}{0.4pt}

    Il \textbf{modello relazionale} è un \hl{modello di dati} che permette di definire le proprietà che la \textit{popolazione} di una base di dati deve rispettare.

    \vario{Proprietà del modello relazionale}{
      Il modello relazionale è \textbf{value-based} (basato sui valori).
      
      \smallskip
      
      Il modello non prevede l’uso di puntatori per rappresentare legami tra i dati. 
      
      Questo implica:
      \begin{itemize}[label=$\diamond$]
        \item completa indipendenza dall’implementazione e dallo schema fisico
        \smallskip
        \item facilità di trasferimento dei dati tra sistemi
        \smallskip
        \item viene rappresentato solo ciò che è rilevante per il sistema informativo
      \end{itemize}
    }
    
    I \underline{costrutti principali} del modello relazionale sono:
    \begin{itemize}[label=$\triangleright$]
      \item \textcolor{azzurrino}{\textbf{domini di base}}: \hl{insieme di valori atomici} che rappresentano i tipi di dati elementari
      \medskip
      \item \textcolor{azzurrino}{\textbf{relazione (o tabella)}}: rappresenta un \hl{insieme di entità o di relazioni} tra entità.
      
      Possono essere definite come:
      \begin{itemize}[label=$\star$]
        \item insieme di \textbf{ennuple}
        \item insieme di \textbf{tuple}
      \end{itemize}
      \medskip
      \item \textcolor{azzurrino}{\textbf{superchiavi}}, \textcolor{azzurrino}{\textbf{chiavi}} e \textcolor{azzurrino}{\textbf{chiavi primarie}}: \hl{insieme di attributi} che identificano univocamente ogni ennupla di una relazione
      \medskip
      \item \textcolor{azzurrino}{\textbf{vincoli di integrità referenziale}}: \hl{vincoli} che garantiscono la coerenza tra i dati memorizzati in relazioni diverse
      \medskip
      \item \textcolor{azzurrino}{\textbf{vincoli di integrità generici}}: \hl{vincoli} che garantiscono la coerenza dei dati memorizzati in una singola relazione
    \end{itemize}

    \definizione{domini di base}{I \textcolor{azzurrino}{\textbf{domini di base}} sono i \hl{domini da cui si scelgono i valori delle proprietà} delle istanze di informazione da rappresentare.

    \smallskip
    I domini tipici sono:
    \begin{itemize}[label=$\diamond$]
      \item \textit{caratteri}
      \item \textit{stringhe di caratteri}
      \item \textit{numeri interi}
      \item \textit{numeri deciamali a virgola fissa}
      \item \textit{numeri decimali a virgola mobile}
      \item \textit{domini del tempo}: rappresentano istanti e intervalli di tempo
    \end{itemize}}

    \definizione{relazione}{
      Una \textcolor{azzurrino}{\textbf{relazione}} può essere vista come una tabella.

      \begin{figure}[H]
        \centering
        \includegraphics[width=0.4\textwidth]{images/relazioneComeTabella.png}
      \end{figure}

      Una tabella è un contenitore di dati la cui struttura è caratterizzata da una lista di colonne:
      \begin{itemize}
        \item ogni riga descrive le caratteristiche di un’istanza dell’informazione da rappresentare
        \smallskip
        \item valori contenuti nelle colonne descrivono sempre la stessa proprietà delle istanze di informazione da rappresentare
      \end{itemize}
    }

    \definizione{relazione (insieme di tuple - Lists)}{
      Dati n insiemi di valori (\textit{domini}) $\mathrm{D_1}, \dots , \mathrm{D_n}$, con n > 0 e indicato con $\mathrm{D_1} \times \dots \times \mathrm{D_n}$ il loro prodotto cartesiano:
      \begin{equation*}
        \mathrm{D_1} \times \dots \times \mathrm{D_n} = {(\mathrm{v_1}, \dots , \mathrm{v_n}) \ | \ \mathrm{v_1} \in \mathrm{D_1} \land \dots \land \mathrm{v_n} \in \mathrm{D_n}}
      \end{equation*}

      una \textcolor{azzurrino}{relazione $\rho$ di grado n} è un qualsiasi sottoinsieme di $\mathrm{D_1} \times \dots \times \mathrm{D_n}$:
      \begin{equation*}
        \rho \subseteq \mathrm{D_1} \times \dots \times \mathrm{D_n}
      \end{equation*}

      dove ($\mathrm{v_1}, \dots , \mathrm{v_n}$) è una ennupla della relazione e |$\rho$| è la cardinalità della relazione (\textit{numero di ennuple}).

      \noindent\rule{\linewidth}{0.4pt}

      \textbf{N.B.}
      \begin{itemize}[label=$\triangleright$]
        \item domini $\mathrm{D_1}, \dots , \mathrm{D_n}$ possono essere a cardinalità infinita, mentre le relazioni sono sempre a cardinalità finita
        \smallskip
        \item non è definito alcun ordinamento sulle ennuple di una relazione
        \smallskip
        \item non sono ammessi duplicati di una ennupla
        \smallskip
        \item nella definizione di relazione come insieme di ennuple, i valori delle ennuple sono ordinati
      \end{itemize}
    }

    \esempio{relazione come insieme di tuple (Lists)}{
      Consideriamo la tabella:
      \begin{figure}[H]
        \centering
        \includegraphics[width=0.4\textwidth]{images/relazioneComeTabella.png}
      \end{figure}

      La relazione può essere vista come insieme di ennuple (relazione città):
      \[
        \rho =
        \left\{
        \begin{array}{l}
        (\mathrm{Milano}, 20100, 1300000) \\
        (\mathrm{Verona}, 37100, 350000) \\
        (\mathrm{Brescia}, 25100, 250000)
        \end{array}
        \right\}
      \]

      dove:
      \begin{equation*}
        \rho \subseteq \mathrm{D_1} \times \mathrm{D_2} \times \mathrm{D_3}
      \end{equation*}
    }

    \textbf{Accesso ai valori di una tupla}

    Se t è una ennupla ($\mathrm{v_1}, \dots , \mathrm{v_n}$) il valore posto in i-esima posizione si indica con la notazione:
    \begin{equation*}
      \boldsymbol{\mathrm{t[i]}}
    \end{equation*}

    Questa modalità di accesso ai valori però non è efficace per l’uso pratico delle relazioni.\\[0.2ex]
    Si preferisce, quindi, assegnare un nome alle colonne.

    \definizione{relazione (insime di tuple - Mappings)}{
      Sia $\mathrm{X}$ un insieme di nomi e sia $\Delta$ l’insieme di tutti i domini di base ammessi dal modello.
      
      \smallskip
      
      Si definisce la funzione:
      \begin{equation*}
        \boldsymbol{\mathrm{DOM} : \mathrm{X} \rightarrow \Delta}
      \end{equation*}

      Tale funzione associa ad ogni nome $\mathrm{A} \in \mathrm{X}$ un dominio $\mathrm{DOM(A)} \in \Delta$.
      
      \smallskip
      
      I \hl{nomi di $\mathrm{X}$ si definiscono attributi}.
      
      \smallskip
      
      Una tupla t su $\mathrm{X}$ è una funzione:
      \begin{equation*}
        \mathrm{t} : \mathrm{X} \rightarrow \bigcup_{\mathrm{A} \ \in \ \mathrm{X}} \mathrm{DOM(A)}
      \end{equation*}

      dove:
      \begin{equation*}
        \mathrm{t[A}] = \mathrm{v} \in \mathrm{DOM(A)}
      \end{equation*}

      \noindent\rule{\linewidth}{0.4pt}

      \textbf{N.B.}
      \begin{itemize}[label=$\triangleright$]
        \item una relazione è un insieme di tuple e quindi non può contenere tuple duplicate
        \smallskip
        \item domini per gli attributi possono essere solo domini di base, non sono ammessi altri domini (non sono ammessi attributi annidati)
        \smallskip
        \item una base di dati relazionale è composta da più relazioni
      \end{itemize}
    }

    \esempio{relazione come insieme di tuple (Mappings)}{
      Consideriamo la tabella:
      \begin{figure}[H]
        \centering
        \includegraphics[width=0.4\textwidth]{images/relazioneComeTabella.png}
      \end{figure}

      Definiamo l’insieme di attributi:
      \begin{equation*}
        \mathrm{X = \{nome, CAP, abitanti\}}
      \end{equation*}

      Definiamo il dominio di ciascun attributo:
      \begin{description}
        \centering{
          \item $\mathrm{DOM(nome) = stringhe \ di \ caratteri}$
          \smallskip
          \item $\mathrm{DOM(CAP) = numeri \ interi}$
          \smallskip
          \item $\mathrm{DOM(abitanti) = numeri \ interi}$ 
        }
      \end{description}


      \medskip
      La tabella della relazione è rappresentata come un insieme di tuple:
      \begin{equation*}
        \rho_{\mathrm{X}} = \{\mathrm{t_1, t_2, t_3}\}
      \end{equation*}

      dove:
      \begin{description}
        \centering{
          \item $\mathrm{t_1[nome] = Milano \quad t_1[CAP] = 20100 \quad t_1[abitanti] = 1300000}$
          \smallskip
          \item $\mathrm{t_2[nome] = Verona \quad t_2[CAP] = 37100 \quad t_2[abitanti] = 350000}$
          \smallskip
          \item $\mathrm{t_3[nome] = Brescia \quad t_3[CAP] = 25100 \quad t_3[abitanti] = 250000}$ 
        }
      \end{description}
    }


    \section{Progettazione nel modello relazionale}
    Nel modello relazione lo schema è costituito da un insieme di relazione (\textit{tabelle}).

    Le dipendenze tra proprietà elementari (\textit{attributi}) sono costituite dalle dipendenze funzionali.

    \vario{Requisiti da implementare nel modello relazionale}{
      Si vuole rappresentare in una base di dati relazionale le informazioni sulla proprietà degli appartamenti siti nel comune di Verona.

      \smallskip

      Per ogni appartamento viene memorizzato: \textit{codice ecografico} (univoco), \textit{categoria catastale} e \textit{indirizzo} (via, numero civico, ubalterno).

      \smallskip
      
      Per ogni proprietario si registra: \textit{codice fiscale}, \textit{nome}, \textit{cognome} e \textit{data di nascita}.

      \smallskip
      
      Un appartamento può essere in comproprietà e un proprietario può possedere più appartamenti.
    }

    \subsection{Relazione unica}
    Una soluzione possibile è rappresentata da un’unica relazione che contiene tutte le informazioni sugli appartamenti e sui proprietari:

    \begin{figure}[H]
      \centering
      \includegraphics[width=0.6\textwidth]{images/relazioneUnica.png}
      \caption{schema relazionale con una sola relazione Proprietà}
    \end{figure}

    Questa tabella associa ogni appartamento ad un proprietario perchè i dati dell’appartamento e della persona fanno parte della stessa riga. 
    
    \vario{Problematiche}{
      Il problema è che non c’è distinzione tra le due categorie di informazioni. 
      
      \smallskip
      
      Questo porta a diversi problemi:
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{mancata separazione delle informazioni}: le due categorie di informazione non possono esistere separatamente
        \smallskip
        \item \textbf{ridondanza dei dati}: se un proprietario possiede più appartamenti i suoi dati vengono memorizzati più volte. 
        
        Questo comporta un aumento dello spazio di memorizzazione e una possibile inconsistenza dello stesso dato.
      \end{itemize}
    }

    
    Le varie problematiche sono descritte in modo più dettagliato come \textbf{anomalie} e si dividono in:
    \begin{itemize}[label=$\circ$]
      \item \textcolor{azzurrino}{\textbf{anomalia di aggiornamento}}: per aggiornare il valore di un attributo bisogna modificare tale valore su più tuple
      \smallskip
      \item \textcolor{azzurrino}{\textbf{anomalie di inserimento}}: per inserire una nuova tupla è necessario inserire valori al momento sconosciuti (sostituibili da valori nulli) per gli attributi non disponibili
      \smallskip
      \item \textcolor{azzurrino}{\textbf{anomalie di cancellazione}}: per cancellare una tupla è necessario cancellare valori ancola validi oppure inserire valori nulli per gli attributi da cancellare
    \end{itemize}

    \section{Valori nulli}
    Dalla definizione di relazione, si ha che ogni tupla deve sempre contenere nei propri attributi valori significativi appartenenti ai domini di base del modello.\\[0.2ex]
    Non sempre, però, in una base di dati reale esistono i valori per tutti gli attributi di una tupla.

    Un valore può mancare nei seguenti casi:
    \begin{itemize}[label=$\diamond$]
      \item \textbf{valore di un attributo A inesistente} (attributo opzionale a livello concettuale): non esiste per questa tupla un valore per l’attributo A
      \smallskip
      \item \textbf{valore di un attributo A è sconosciuto}: esiste un valore per l’attributo A di questa tupla ma non è noto alla base di dati
      \smallskip
      \item \textbf{valore di un attributo A è inesistente o sconosciuto}
    \end{itemize}
    
    Per poter gestire tali situazioni viene introdotto nel modello relazionale un valore speciale detto \textcolor{azzurrino}{\textbf{valore nullo}}: gli attributi di una tupla possono assumere un valore del dominio oppure il valore nullo.

    \definizione{tupla con valore nullo}{
      Una tupla su $\mathrm{X}$ con valore nullo è definita come una funzione:
      \begin{equation*}
        \boldsymbol{\mathrm{t} : \mathrm{X} \rightarrow \{\mathrm{NULL}\} \cup (\bigcup_{\mathrm{A} \ \in \ \mathrm{X}} \mathrm{DOM(A)})}
      \end{equation*}

      dove:
      \begin{equation*}
        \mathrm{t[A]} = \mathrm{v} \in \mathrm{DOM(A)} \lor \mathrm{t[A]} = \mathrm{NULL}
      \end{equation*}
    }

    \vario{Osservazioni}{
      La presenza di valori nulli è accettabile solo in alcuni attributi (non è possibile avere tuple di soli valori nulli).
      
      In particolare, negli attributi replicati per rappresentare legami tra tuple la presenza di valori nulli può rendere inutilizzabile l’informazione.
    }

    \esempio{schema relazionale}{
      Rappresentare attraverso un insieme di relazioni le informazioni contenute in un oriario ferroviario, dove si riportano per ogni treno in partenza dalla stazione di Verona Porta Nuova: \textit{numero}, \textit{orario di partenza}, \textit{destinazione finale}, \textit{categoria}, \textit{fermate intermedie} (con orario di fermata) e \textit{orario di arrivo} alla destinazione finale.

      \smallskip
      Lo schema relazionale è il seguente:
      \begin{equation*}
        \mathrm{TRENO(numero, categoria, oraPartenza, destinazione, oraArrivo)}
      \end{equation*}
      \begin{equation*}
        \mathrm{FERMATA(numTreno, stazione, orario)}
      \end{equation*}

      In progettazione concettuale equivale a due entità (Treno e Stazione) collegate da una relazione (Fermata).
    }

    \section{Vincoli d'integrità}
    \definizione{vincolo d'integrità}{
      Un \textcolor{azzurrino}{\textbf{vincolo di integrità}} è una \hl{condizione}, espressa da un predicato, che deve essere sempre soddisfatta da ogni istanza della base di dati.
    }

    \subsection{Vincoli di dominio}
    Impongono una \hl{restrizione sui valori che un attributo può assumere}.

    \esempio{vincolo di dominio}{
      \begin{itemize}[label=$\triangleright$]
        \item $\forall \mathrm{t} \in \mathrm{TRENO} : \mathrm{t[oraPartenza]} \in \{0,1, \dots, 23\}$
        \smallskip
        \item $\forall \mathrm{t} \in \mathrm{TRENO} : \mathrm{t[minutoPartenza]} \in \{0,1, \dots, 59\}$
        \smallskip
        \item $\forall \mathrm{t} \in \mathrm{TRENO} : \mathrm{t[numero]} > 0$
      \end{itemize}
    }

    \subsection{Vincoli di tupla}
    Impongono una \hl{restrizione alla combinazione di valori che una tupla della relazione può assumere} indipendentemente dalle altre tuple.

    \esempio{vincolo di tupla}{
      \begin{itemize}[label=$\diamond$]
        \item $\forall \mathrm{t} \in \mathrm{TRENO} : \mathrm{t[numero]} > 500 \Rightarrow \mathrm{t[categoria]}$ = 'regionale'
      \end{itemize}
    }

    \subsection{Vincoli di intrarelazionali}
    Impongono una \hl{restrizione al contenuto di una relazione} e specificano una \hl{condizione che ogni tupla della relazione deve soddisfare} rispetto alle altre tuple della medesima relazione.

    Una sottocateoria importante di questi vincoli include i \textbf{vincoli di chiave}.

    \definizione{superchiave}{
      Data una relazione di schema $\mathrm{R(X)}$, un insieme di attributi $\mathrm{K}$ sottoinsieme di $\mathrm{X}$ è \textcolor{azzurrino}{\textbf{superchiave}} per $\mathrm{R(X)}$ se per ogni istanza r di $\mathrm{R(X)}$ vale la seguente condizione:
      \begin{equation*}
        \boldsymbol{\forall \mathrm{t}, \mathrm{t'} \in \mathrm{r}: \mathrm{t} \neq \mathrm{t'} \Rightarrow \mathrm{t[K]} \neq \mathrm{t'[K]}}
      \end{equation*}

      dove:
      \begin{equation*}
        \mathrm{t[K]} \neq \mathrm{t'[K]} \equiv \exists \mathrm{A_i} \in \mathrm{K}: \mathrm{t[A_i]} \neq \mathrm{t'[A_i]}
      \end{equation*}
    }

    \definizione{chiave candidata}{
      Data una relazione di schema R(X), un insieme di attributi K sottoinsieme di X è chiave candidata (o chiave) per R(X) se K è superchiave per R(X) e vale la seguente condizione:
      \begin{equation*}
        \boldsymbol{\neg \forall \mathrm{K'} \subseteq \mathrm{K}: \mathrm{K'}} \ \ \text{è superchiave per } \mathrm{R(X)}
      \end{equation*}
    }

    \definizione{chiave primaria}{
    Data una relazione di schema $\mathrm{R(X)}$ la sua chiave primaria è la chiave candidata scelta per identificare le tuple della relazione.\\[0.2ex]
    La chiave primaria viene usata per rappresentare i legami tra le relazioni.

    \noindent\rule{\linewidth}{0.4pt}

    \textbf{Caratteristiche}
    \begin{itemize}[label=$\circ$]
      \item non contiene \textbf{mai} valori nulli
      \smallskip
      \item su $\mathrm{K}$ il sistema genera una struttura d’accesso ai dati (o indice) per supportare le interrogazioni
    \end{itemize}
    }

    La chiave primaria di una relazione $\mathrm{R(X)}$ viene indicata sottolineando gli attributi della chiave.

    \esempio{notazione della chiave primaria}{
      La chiave primaria della tabella \textit{Treno} è l’attributo numero:
      \begin{description}
        \centering
        \item TRENO(\underline{numero}, categoria, oraPartenza, destinazione, oraArrivo)
      \end{description}

      \medskip
      La chiave primaria della tabella \textit{Fermata} è:
      \begin{description}
        \centering
        \item FERMATA(\underline{numTreno, stazione}, orario) 
      \end{description}
    }

    \esempio{vincoli di chiave}{
      Consideriamo la tabella Treno.
      
      \smallskip
       I seguenti sottoinsiemi di $\mathrm{X}$ sono:
       \begin{itemize}
        \item $\mathrm{K1} = \{\mathrm{numero}\}$: superchiave e chiave candidata
        \smallskip
        \item $\mathrm{K2}$ = $\{\mathrm{numero}, \mathrm{categoria}\}$: superchiave, ma non chiave candidata
        \smallskip
        \item $\mathrm{K3} = \{\mathrm{oraPartenza}, \mathrm{minutoPartenzart}, \mathrm{destinazione}, \mathrm{categoria}\}$: superchiave e chiave candidata
       \end{itemize}
    }

    \subsection{Vincoli interrazionali}
    Impongono una \hl{restrizione al contenuto di una relazione} e specificano una \hl{condizione che ogni tupla della relazione deve soddisfare} rispetto alle tuple di altre relazioni della base di dati.

    Una sottocategoria importante di tali vincoli include i \textbf{vincoli d'integrità referenziale} (o vincoli sulle chiavi esportate).

    \definizione{vincoli d'integrità referenziale}{
      Un vincolo di integrità referenziale tra un insieme di attributi $\mathrm{Y} = \{\mathrm{A_1}, . . . , \mathrm{A_p}\}$ di $\mathrm{R_1}$ e un insieme di attributi $\mathrm{K} = \{\mathrm{K_1}, . . . , \mathrm{K_p}\}$, chiave primaria di un’altra relazione $\mathrm{R_2}$, è soddisfatto se, per ogni istanza $\mathrm{r_1}$ di $\mathrm{R_1}$ e per ogni istanza $\mathrm{r_2}$ di $\mathrm{R_2}$ vale la seguente condizione:
      \begin{equation*}
        \boldsymbol{\forall \mathrm{t} \in \mathrm{r_1}: \exists \mathrm{s} \in \mathrm{r_2}: \forall \mathrm{i} \in \{1, \dots, \mathrm{p}\}: \mathrm{t[A_i]} = \mathrm{s[K_i]}}
      \end{equation*}

      In presenza di valori nulli (legame opzionale) la condizione diventa:
      \begin{equation*}
        \begin{aligned}
        \boldsymbol{\forall \mathrm{t} \in \mathrm{r_1}:} \;&
        \boldsymbol{\exists \mathrm{s} \in \mathrm{r_2}:} \\
        &\boldsymbol{\Big(
        (\forall \mathrm{i} \in \{1, \dots, \mathrm{p}\}:
        \mathrm{t[A_i]} = \mathrm{s[K_i]})} \\
        &\boldsymbol{\lor
        (\exists \mathrm{i} \in \{1, \dots, \mathrm{p}\}:
        \mathrm{t[A_i]} = \mathrm{NULL})
        \Big)}
        \end{aligned}
      \end{equation*}
    }

    Il vincolo di integrità referenziale viene indicato in due modi.

    \vario{Metodo 1}{
      Riquadrando gli attributi soggetti al vincolo e collegando con una freccia il riquadro alla relazione (tabella) da cui la chiave proviene.

      \noindent\rule{\linewidth}{0.4pt}

      \smallskip
      \textbf{Esempio}

      Supponendo che esista un vincolo di integrità referenziale sull’attributo numTreno della tabella \textit{Fermata} rispetto alla tabella \textit{Treno}, la notazione per indicarlo è la seguente:
      \begin{figure}[H]
        \centering
        \includegraphics[width=0.5\textwidth]{images/vincoloIntegritàReferenziale.png}
      \end{figure}

    }

    \vario{Metodo 2}{
      Indicare con una freccia l’attributo soggetto al vincolo verso la relazione da cui la chiave proviene, ad esempio:
      \begin{description}
        \centering
        \item $\mathrm{R_1}, \mathrm{D} \rightarrow \mathrm{R_2}$
        \smallskip
        \item $\mathrm{R_3(Y, X)} \rightarrow \mathrm{R_1}$
      \end{description}

      \smallskip
      L’attributo opzionale è indicato con un asterisco accanto al nome dell’attributo.
    }

    La ristrutturazione dello schema concettuale può essere necessaria per i seguenti motivi:

      \item \textcolor{azzurrino}{\textbf{semplificare la fase successiva di traduzione}}

    L’obiettivo dell’analisi delle ridondanze è quello di analizzare i dati derivabili presenti nello schema e decidere se:
    \begin{itemize}[label=$\star$]
      \item mantenere il dato derivabile (memorizzandolo esplicitamente)
      \smallskip
      \item calcolare il dato derivabile quando serve
    \end{itemize}

    \esempio{tipologie di dati derivabili}{
      \begin{itemize}[label=$\triangleright$]
        \item attributi derivabili da altri attributi della stessa entità (o relazione)
        \smallskip
        \item attributi derivabili da attributi di altre entità (o relazioni)
        \smallskip
        \item attributi derivabili dal conteggio di occorrenze
      \end{itemize}
    }

    \subsubsection{Eliminazione delle generalizzazioni}
    L’obiettivo è quello di sostituire le generalizzazioni presenti nello schema E-R con strutture alternative, in quanto non direttamente rappresentabili nel modello relazionale.

        \smallskip
        \item accorpamento nel padre delle relazioni che coinvolgono le entità figlie assegnando cardinalità minima uguale a zero (lato entità padre)
        \smallskip
        \item aggiunta di un attributo "tipo" per distinguere nel padre le occorrenze delle entità figlie eliminate
       \end{enumerate}

        \item eliminazione dell’entità padre
        \smallskip
        \item replicazione degli attributi dell’entità padre su ogni entità figlia
        \smallskip
        \item partizionamento sui figli delle relazioni che coinvolgono l’entità padre assegnando cardinalità minima uguale a zero (lato entità esterne)

    \subsubsection{Accorpamento (o partizionamento) di entità e relazioni}
    Il \textcolor{azzurrino}{\textbf{partizionamento di entità}} è la sostituzione di un’entità dello schema con una coppia di entità con lo stesso identificatore e una relazione uno a uno che le lega tra loro.

    Si esegue solo nel caso in cui esistano operazioni frequenti che trattano solo un sottoinsieme degli attributi dell’entità da partizionare.

    \medskip
    Il \textcolor{azzurrino}{\textbf{partizionamento di relazioni}} si realizza sostituendo una relazione R con due relazioni distinte $\mathrm{R_1}$ e $\mathrm{R_2}$, dove le occorrenze di R si partizionano tra $\mathrm{R_1}$ e $\mathrm{R_2}$.\\[0.2ex] 
    È conveniente quando esistono operazioni che si riferiscono ad un sottoinsieme delle occorrenze di R e altre operazioni che si riferiscono alle altre occorrenze di R.

    \subsubsection{Scelta degli identificatori principali}{
      Va scelto un identificatore principale per ogni entità in quanto tale identificatore verrà utilizzato nel modello relazionale per identificare le tuple che rappresentano istanze di entità.
      
      Si scelgono seguendo i seguenti criteri:
      \begin{itemize}[label=$\triangleright$]
        \item gli identificatori che includono attributi opzionali non possono essere chiavi primarie
        \smallskip
        \item meglio identificatori con pochi attributi
        \smallskip
        \item meglio identificatori utilizzati nelle operazioni
      \end{itemize}
    }

    \end{itemize}

    \chapter{Tecnologie NoSQL}
    I vari sistemi NoSQL sono molto diversi tra loro in termini di rappresentazione dei dati e di linguaggi per la loro manipolazione.\\[0.2ex]
    Nonostante questa diversificazione delle soluzioni attualmente disponibili, è possibile classificare la maggior parte dei sistemi NoSQL in alcune categorie principali:

    \begin{description}[labelindent=0.5cm]
      \item[\textcolor{azzurrino}{sistemi chiave-valore}:] \hl{ogni dato è costituito da una coppia $(chiave, valore)$}.

      \hspace*{0.5cm}\textcolor{red}{\textbf{chiave}}: identifica le istanze

      \smallskip
      \hspace*{0.5cm}\textcolor{red}{\textbf{valore}}: qualsiasi struttura (anche non omogenea)

      \smallskip
      La principale modalità di accesso offerta è sulla base di un valore per la chiave.

      Il campo \textit{valore} associato a una \textit{chiave} è tipicamente opaco per l'utente e non esistono meccanismi di ricerca basati sul suo contenuto.

      Non esiste, quindi, uno schema a cui riferirsi e le primitive di accesso ai dati sono in genere molto semplici.

      \smallskip
      $\Rightarrow$ \textbf{Hadoop}

      \bigskip
      \item[\textcolor{azzurrino}{sistemi basati su documenti (document store)}:]
      \hl{dati rappresentati sotto forma di documento strutturato e nidificato},
      tipicamente utilizzando il formato standard JSON.

      Ogni oggetto della collezione ha una struttura complessa senza schema fisso.

      Consentono la definizione di indici secondari su proprietà degli oggetti.

      Le primitive di accesso ai dati consentono di interrogare una base di dati sulla base dei campi e sottocampi di questi documenti.

      \smallskip
      $\Rightarrow$ \textbf{MongoDB}, \textbf{CouchBase}

      \bigskip
      \item[\textcolor{azzurrino}{sistemi basati su colonne (column store)}:] \hl{dati sono raggruppati in record} (\textit{righe}), fatti da una serie di campi (\textit{colonne})
      la cui quantità e tipologia può variare da record a record.

      L'accesso ai dati avviene specificando le colonne di interesse (proiezione) e/o valori per queste colonne (selezione).

      La rappresentazione è quindi affine al modello relazionale (righe e colonne), ma senza alcun vincolo sulla struttura delle righe.

      \smallskip
      $\Rightarrow$ \textbf{BigTable} di Google, \textbf{HBase}, \textbf{Cassandra}

      \bigskip
      \item[\textcolor{azzurrino}{sistemi a grafo (graph store)}:]
      \hl{i dati vengono rappresentati in forma di grafo}.

      \smallskip
      \hspace*{0.5cm}\textcolor{red}{\textbf{nodi}}: rappresentano gli oggetti

      \smallskip
      \hspace*{0.5cm}\textcolor{red}{\textbf{archi}}: rappresentano le relazioni tra questi o proprietà

      \smallskip
      Le operazioni di accesso ai dati si basano su primitive di navigazione nel grafo a partire da uno o più nodi iniziali.

      Si tratta di sistemi particolarmente adatti quando si vuole gestire una base di dati che ha una naturale rappresentazione mediante un grafo.

      \smallskip
      $\Rightarrow$ \textbf{Neo4j}, \textbf{OrientDB}

    \end{description}


    \section{Sistemi basati su documenti (document store)}

    \subsection{Caratteristiche principali}
    Nei sistemi basati su documenti, gli oggetti memorizzati in una base di dati possono avere anche una \textcolor{azzurrino}{\textbf{struttura complessa}} (no tuple piatte) e possono essere acceduti facendo rierimento a componenti interne dell'oggetto, senza necessariamente coinvolgere la chiave primaria.

    I meccanismi per gestire e manipolare i dati offerti sono più flessibili, pur garantendo elevate qualità di efficienza e scalabilità su infrastutture basate su cluster di computer.

    I dati sono organizzati in collezioni di \textcolor{azzurrino}{\textbf{informazioni omogenee}} per le quali però non esite un rigido schema di riferimento.\\[0.2ex]
    Ogni \textcolor{azzurrino}{\textbf{collezioni}} contiene \textcolor{azzurrino}{\textbf{oggetti}} (\textcolor{azzurrino}{\textit{documenti}}) che sono rappresentati utilizzando tipicamente un formato "auto-descritttivo" standard (Json), possono avere una struttura complessa di tipo gerarchico e sono concettualemnete \textcolor{azzurrino}{\textbf{simili}}, \textcolor{azzurrino}{\textbf{ma non identici tra loro}}.

    \subsection{Progettazione dei dati come documenti}

    La decisione importante nel processo di progettazione dei dati usando questo nuovo approcio riguarda:
    \begin{itemize}
      \item scelta della \underline{chiave di accesso}
      \item \underline{struttura dei documenti} (\textit{grado di incapsulamento})
      \item \underline{rappresentazione delle relazioni} (\textit{legami}) tra documenti
    \end{itemize}

    Esistono due approci per rappresentare le relazioni tra documenti:
    \begin{enumerate}
      \item \textbf{\textcolor{azzurrino}{tramite riferimenti}}: si ottengono esportando l'ID del documento
      \item \textbf{\textcolor{azzurrino}{tramite documenti nidificati}} (\textit{incapsulamento})
    \end{enumerate}

    L'\hl{\textbf{uso di riferimenti} per rappresentare i legami tra documenti porta a schemi normalizzati}.

    L'\hl{\textbf{uso di documenti nidificati} porta a schemi non normalizzati e parzialmente rindondanti}, ma molto efficienti in lettura.

        \end{subfigure}
    \end{figure}

    L'\textcolor{azzurrino}{\textbf{approcio normalizzato}} consente di rappresentare il dato in modo simile a quanto avviene nel modello relazionale: \hl{non c'è rindondanza} e gli \hl{aggiornamenti sono più semplici}.

    L'\textcolor{azzurrino}{\textbf{approcio embedded}} consente di rappresen in un unico documento quanto viene rappresentato di solito nel modello E-R con un'entità E insieme alle sue "entità deboli" (entità collegate ad E attraverso identificatori esterni).\\[0.2ex]
    E' posssibile rappresentare nello stesso documento un'istanza di informazione con altre istanze in essa logicamente contenute.

    \subsection{Traduzione da schema E-R a collezione di documenti}
    Supponiamo di applicare un \textbf{approcio embedded} a uno schema E-R già ristrutturato (privo di generalizzazioni).\\[0.2ex]
    Le relazioni ternarie (o ennarie) devono essere eliminate.\\[0.2ex]
    A partire dalle operazioni di elaborazione da eseguire sui dati vengono identificate le \textbf{collezioni di documenti da generare nel sistema}:
    \begin{equation*}
      \{\mathrm{DOC_1}, \dots, \mathrm{DOC_N}\}
    \end{equation*}
    Possono essere al massimo una per ogni entità dello schema.
    
    \vario{Regole di base}{
      \begin{itemize}[label=$\diamond$]
        \item ogni collezione $\mathrm{DOC_i}$ è in corrispondenza con un'entità EP dello schema E-R, dettà entità principale della collezione di documenti
        \smallskip
        \item ogni istanza delle entità non principali sarà rappresentato da uno o più documenti incapsulato in un altro documento che rapppresenta un'istanza di un'entità principale
        \smallskip
        \item le istanze di relazione saranno rapppresentate strutturalmente oppure esportando ID di documento o chiavi primarie
      \end{itemize}
    }

    \chapter{Operazioni su una base di dati relazionale}
    I linguaggi per gestire i dati sono divisi in due categorie a seconda del tipo di operazioni che consentono di eseguire:
    \begin{itemize}[label=$\diamond$]
      \item \textcolor{azzurrino}{\textbf{Data Definition Language (DDL)}}: \hl{usato per la \textbf{definizione} dei dati}.
      
      Questo linguaggio consente di:
      \begin{itemize}[label=$\circ$]
        \item creare lo schema della base di dati
        \smallskip
        \item modificare lo schema della base di dati
      \end{itemize}
      \medskip
      \item \textcolor{azzurrino}{\textbf{Data Manipulation Language (DML)}}: \hl{usato per la \textbf{manipolazione} dei dati}.
      
      Questo linguaggio consente di:
      \begin{itemize}[label=$\circ$]
        \item inserire o cancellare tuple nelle relazioni
        \smallskip
        \item aggiornare i valori nelle tuple delle relazioni
        \smallskip
        \item interrogare le relazioni per estrarre informazioni
      \end{itemize}
    \end{itemize}

    \medskip

    I linguaggi di interrogazione che fanno parte del DML si dividono in due categorie:
    \begin{itemize}[label=$\triangleright$]
      \item \textbf{linguaggi procedurali}: specificano il procedimento per ottenere il risultato (es. algebra relazionale)
      \medskip
      \item \textbf{linguaggi dichiarativi}: specificano le proprietà del risultato (es. calcolo relazionale e SQL)
    \end{itemize}

    \section{Algebra relazionale}
    L’\textcolor{azzurrino}{\textbf{algebra relazionale}} è un \hl{linguaggio procedurale per l’interrogazione delle basi di dati} relazionali.\\[0.2ex] 
    Questo linguaggio è un \underline{insieme di operatori su relazioni}.

    \vario{"Segnatura" degli operatori}{
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{operatore unario}
        \begin{equation*}
          \boldsymbol{\mathrm{op_{parametri}(r_1)} \rightarrow \mathrm{r_2}}
        \end{equation*}

        dove:
        \begin{itemize}[label=$\star$]
          \item $\mathrm{r_1}$ e $\mathrm{r_2}$: relazioni
          \item $\mathrm{op}$: operatore con parametri specifici
        \end{itemize}


        \medskip
        \item \textbf{operatore binario}
        \begin{equation*}
          \boldsymbol{\mathrm{r_1 op_{parametri}(r_2)} \rightarrow \mathrm{r_3}}
        \end{equation*}

        dove:
        \begin{itemize}[label=$\star$]
          \item $\mathrm{r_1}$, $\mathrm{r_2}$ e $\mathrm{r_3}$: relazioni
          \item $\mathrm{op}$: operatore con parametri specifici
        \end{itemize}
      \end{itemize}
    }

    Gli operatori si possono classificare in due modi:
    \begin{itemize}[label=$\diamond$]
      \item \textbf{operatori funzionali}:
      \begin{itemize}
        \item operatori insiemistici
        \smallskip
        \item operatori specifici
        \smallskip
        \item operatori di giunzione (o join) tra relazioni
      \end{itemize}
      \medskip
      \item \textbf{operatori derivabili}:
      \begin{itemize}
        \item operatori di base
        \smallskip
        \item operatori derivati
      \end{itemize}
    \end{itemize}

    \subsection{Operatori insiemistici}
    Siccome le relazioni sono insiemi di tuple ha senso applicare ad esse le operazioni insiemistiche.\\[0.2ex]
    Tuttavia va considerato che le relazioni sono insiemi di tuple omogenee, cioè tutte le tuple di una relazione hanno lo stesso schema (stesso insieme di attributi con stessi domini).\\[0.2ex]
    Quindi le operazioni insiemistiche si possono applicare soltanto a relazioni con lo stesso schema.

    Date due relazioni $\mathrm{r_1}$ e $\mathrm{r_2}$ sullo stesso schema $\mathrm{X}$ si definiscono i seguenti operatori.

    \definizione{operatori di base}{
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{unione}:
        \begin{equation*}
          \mathrm{r_1} \cup \mathrm{r_2} = \mathrm{r_3} = \{\mathrm{t} \ | \ \mathrm{t} \in \mathrm{t} \lor \mathrm{t} \in \mathrm{r_2}\}
        \end{equation*}
        \item \textbf{differenza}:
        \begin{equation*}
          \mathrm{r_1} - \mathrm{r_2} = \mathrm{r_3} = \{\mathrm{t} \ | \ \mathrm{t} \in \mathrm{r_1} \land \mathrm{r_1} \mathrm{t} \notin \mathrm{r_2}\}
        \end{equation*}
      \end{itemize}

      \noindent\rule{\linewidth}{0.4pt}

      \smallskip
      \textbf{N.B.}

      $\rightarrow$ unione: tutte le tuple duplicate vengono rappresentate una sola volta

      $\rightarrow$ differenza: non è commutativa
    }

    \definizione{operatori derivati}{
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{intersezione}:
        \begin{equation*}
          \mathrm{r_1} \cap \mathrm{r_2} = \mathrm{r_1} - (\mathrm{r_1} - \mathrm{r_2})
        \end{equation*}

        %foto
      \end{itemize}
    }

    La cardinalità degli operatori insiemistici è il numero di tuple contenute nella relazione risultante:
    \begin{itemize}
      \item \textbf{unione}:
      \begin{equation*}
        \mathrm{max(|r_1|, |r_2|)} \le |\mathrm{r_1} \cup \mathrm{r_2}| \le |\mathrm{r_1}| + |\mathrm{r_2}|
      \end{equation*}
      \smallskip
      \item \textbf{differenza}:
      \begin{equation*}
        0 \le |\mathrm{r_1} - \mathrm{r_2}| \le |r_1|
      \end{equation*}
      \smallskip
      \item \textbf{intersezione}:
      \begin{equation*}
        0 \le |\mathrm{r_1} \cap \mathrm{r_2}| \le \mathrm{min}(|\mathrm{r_1}|, |\mathrm{r_2}|)
      \end{equation*}
    \end{itemize}

    \subsection{Operatori specificici}
    Data una relazione $\mathrm{r_1}$ sullo schema $\mathrm{X} = \{\mathrm{A_1}, \mathrm{A_2}, \dots, \mathrm{A_n}\}$, si definiscono i seguenti operatori (tutti unari).

    \definizione{operatori di base}{
      \begin{itemize}[label=$\triangleright$]
        \item \textbf{ridenominazione}: cambia il nome degli attributi di una relazione.
        
        Sia $\mathrm{Y} = \{\mathrm{B_1}, \mathrm{B_2}, \dots, \mathrm{B_n}\}$ un insieme di attributi con $|\mathrm{Y}| = |\mathrm{X}|$, allora:
        \[
          \rho_{\mathrm{A_1}, \dots, \mathrm{A_n} \rightarrow \mathrm{B_1}, \dots, \mathrm{B_n}}(r_1) = r_2
          = \{\mathrm{t} \mid \exists \mathrm{t}' \in \mathrm{r_1} :
          \forall i \in \{1,\dots,\mathrm{n}\},\;
          \mathrm{t[B_i]} = t\mathrm{'[A_i]}\}
        \]
        \item \textbf{selezione}:consente di estrarre da una relazione le tuple che soddisfano una certa condizione $\mathrm{F(t)}$:
        \begin{equation*}
          \sigma_\mathrm{F}(\mathrm{r_1}) = \mathrm(r_2) = \{\mathrm{t} \mid \mathrm{t} \in \mathrm{r_1} \land F(t)\}
        \end{equation*}

        La condizione F(t) è una formula proposizionale con le seguenti caratteristiche:
        \begin{itemize}[label=$\diamond$]
          \item si ottiene combinando con i connettivi logici $\land$, $\lor$ e $\neg$ condizioni atomiche del tipo $A \ \Theta \ B$ o $A \ \Theta \ c$, dove:
          \begin{itemize}
            \item $\mathrm{A}$, $\mathrm{B}$ $\in \mathrm{X}$
            \item $\Theta \in \{<, =, >, \neq, \ge, \le\}$
            \smallskip
            \item $\mathrm{c} \in \mathrm{DOM(A)}$ è una costante compatibile con il dominio di $\mathrm{A}$
          \end{itemize}
          \medskip
          \item una formula $\mathrm{A} \ \Theta \ \mathrm{B}$ è vera sulla tupla $\mathrm{t}$ se:
          \begin{equation*}
            \mathrm{t[A]} \ \Theta \ \mathrm{t[B]}
          \end{equation*}
          \medskip
          \item una formula $\mathrm{A} \ \Theta \ \mathrm{c}$ è vera sulla tupla $\mathrm{t}$ se:
          \item \begin{equation*}
            \mathrm{t[A]} \ \Theta \ \mathrm{c}
          \end{equation*}
          \medskip
          \item le formule $\mathrm{F_1} \land \mathrm{F_2}$, $\mathrm{F_1} \lor \mathrm{F_2}$ e $\neg \mathrm{F}$ hanno valori booleani definiti dalle note tabelle di verità dei connettivi logici corrispondenti
        \end{itemize}
        \bigskip
        \item \textbf{proiezione}: riduce le tuple di una relazione in verticale, cioè elimina attributi. 
        
        Sia $\mathrm{Y} = \{\mathrm{A_1}, \dots , \mathrm{A_m}\} \subseteq \mathrm{X}$ un sottoinsieme degli attributi di $\mathrm{X}$ allora:
        \begin{equation*}
          \Pi_{\mathrm{Y}}(\mathrm{r_1}) = \mathrm{r_2} = \{\mathrm{t} \mid \exists \mathrm{t'} \in \mathrm{r_1} \ : \ \mathrm{t'[Y]} = \mathrm{t}\}
        \end{equation*}

        dove $\mathrm{t'[Y]}$ indica una tupla $\mathrm{t_0}$ su $\mathrm{Y}$ tale che:
        \begin{equation*}
          \forall \mathrm{A_i} \in \mathrm{Y} \ : \ \mathrm{t_0[A_i]} = \mathrm{t'[A_i]}
        \end{equation*}
      \end{itemize}
    }

    \esempio{opertore di ridenominazione}{
      Consideriamo lo schema relazionale:
      \begin{equation*}
        \mathrm{CAPOLUOGO\_PROV(nome, popolazione, CAP)}
      \end{equation*}
      \begin{equation*}
        \mathrm{COMUNI\_MINORI(nome, abitanti, CAP)}
      \end{equation*}

      Si richiede di produrre la lista di tutti i comuni:
      \begin{equation*}
        \boldsymbol{\rho_{\mathrm{popolazioni} \rightarrow \mathrm{abitanti}}(\mathrm{CAPOLUOGO\_PROV}) \cup \mathrm{COMUNI\_MINORI}}
      \end{equation*}
    }

    \esempio{operatore di selezione}{
      Consideriamo lo schema relazionale:
      \begin{equation*}
        \mathrm{TRENO(\underline{numero}, partOra, partMinuto, cat, dest, arrOra, arrMin)}
      \end{equation*}
      \begin{equation*}
        \mathrm{FERMATA(\underline{numTreno, ora}, min)}
      \end{equation*}

      Si richiede di produrre la lista di tutti i treni non regionali che partono alle 12:00 o dopo le 12:00 e prima delle 16:00
      \begin{equation*}
        \boldsymbol{\mathrm{\sigma_{partOra \ \ge \ 12  \ \land \ partOra \ < \ 16 \ \land cat \ \neq \ 'regionale'}(TRENO)}}
      \end{equation*}
    }

    \esempio{operatore di proiezione}{
      Consideriamo lo schema relazionale:
      \begin{equation*}

      \end{equation*}
      \begin{equation*}
        \mathrm{FERMATA(\underline{numTreno, ora}, min)}
      \end{equation*}

      Si richiede di restituire il numero e l'ora di partenza de treni Frecccia Rossa (cat = 'FR') con destinazione 'Milano Centrale'
      \begin{equation*}
        \boldsymbol{\mathrm{\Pi_{numero, partOra}(\rho_{cat \ = \ FR \ \land \ dest \ = \ 'Milano Centrale'}(TRENO))}}
      \end{equation*}
    }

      \hl{Progettare una base di dati che contenga le informazioni che descrivono la carriera di uno studente del corso di laurea triennale in informatica} dal momento dell'immatricolazione no alla laurea.\\
      In particolare, il sistema registra tutti gli esami sostenuti dagli studenti durante la loro carriera.

      \medskip
      Per ogni \textbf{studente} si memorizzano: nome, cognome, data di nascita, matricola (univoca), data di immatricolazione e data di laurea. 
      
      \medskip
      Per ogni \textbf{insegnamento} erogato si memorizzano: denominazione, anno accademico di erogazione, anno di corso (I, II o III) e docente.
      
      \medskip
      Si vuole tener traccia nella base di dati per ogni studente: 
      \begin{itemize}[label=$\triangleright$]
        \item \textit{insegnamenti frequentati}: indicando percentuale di presenza (> 0) di uno studente alle lezioni
        \smallskip
        \item \textit{esami sostenuti}: indicando data dell'esame e voto finale
      \end{itemize}

      \medskip
      Si vuole, inoltre, poter calcolare quanti studenti hanno frequentato un insegnamento.
    }

      \medskip
      Per ogni conto corrente si deve produrre il saldo.\\
      Si supponga che un cliente possa aprire conti diversi in più filiali della banca.
    }
