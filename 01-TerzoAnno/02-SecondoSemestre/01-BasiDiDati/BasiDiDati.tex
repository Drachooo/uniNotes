    \section{Tecnologia dei DBMS}
    Un DBMS \textbf{(Database Management System)} o sistema di gestione di basi di dati è un sistema software in grado di gestire collezioni di dati che siano e \textbf{persistenti} assicurando la loro \textbf{affidabilità} e \textbf{privatezza}.

    Un DBMS in quanto sistema software deve essere efficiente ed efficace e deve inoltre estendere le funzionalità del file system.
    Esempio: Gestione in memoria secondaria collezioni di dati.


    \definizione{Basi di dati}{Una base di dati è una collezione di dati gestita da un DBMS.}

      \item Architettura di un DBMS:
      \begin{itemize}

        \item Gestore delle interrogazioni: ottimizzazione delle interrogazioni
        \item Gestore dei metodi di accesso: indici primari e secondari, indici B+-tree e indici Hash
        \item Gestore del buffer: interazione tra memoria secondaria e memoria centrale

        \item Gestore della concorrenza: gestione dell’esecuzione concorrente (VSR, CSR, locking a due fasi)
        \item Gestore dell’affidabilità: gestione dei guasti
        \end{itemize}
      \item Architettura di un sistema document-based
      
    \end{itemize}


    \subsection{Transazione}

    \definizione{Transazione}{È \textcolor{red}{un’unità di lavoro} svolta da un programma applicativo (che interagisce con una base di dati) per la quale si vogliono garantire proprietà di correttezza, robustezza e isolamento.}

    Le caratteristiche principali sono:
    \begin{itemize}
      \item Una transazione o va a buon fine e ha effetto sulla base di dati o abortisce e non ha nessun effetto sulla base di dati.
      \item \textcolor{red}{O tutto o niente!} Non sono ammesse esecuzioni parziali
    \end{itemize}


    Sintassi per definire una transazione in SQL:\\
    
    \textcolor{purple}{<transazione> $\rightarrow$ begin transaction}\par
    \textcolor{purple}{\hspace{8em}<programma>}\par
    \textcolor{purple}{\hspace{8em}end transaction}\\
   \textcolor{purple} {<programma> $\rightarrow$ $\langle$ <istruzione> | commit work | rollback work $\rangle$\\}
    \hspace{8em}\textcolor{purple}{\{ $\langle$ <istruzione> | commit work | rollback work $\rangle$\}}

    La transazione va a buon fine all’esecuzione di un commit work ma non ha alcun effetto se viene eseguito un rollback work.
    
    Una transazione è ben formata se:
    \begin{itemize}
      \item Inizia con un begin transaction.
      \item Termina con un end transaction.
      \item La sua esecuzione comporta il raggiungimento di un commit o di un rollback work e dopo il commit/rollback non si eseguono altri accessi alla base di dati.
    \end{itemize}

    Esempio di transazione ben formata:\\
    \textcolor{cyan}{begin transaction}\\
    \textcolor{cyan}{\hspace{2em}update CONTO set saldo = saldo - 1200}\\
    \textcolor{cyan}{\hspace{4em}where filiale = '005' and numero = 15;}\\
    \textcolor{cyan}{\hspace{2em}update CONTO set saldo = saldo + 1200}\\
    \textcolor{cyan}{\hspace{4em}where filiale = '005' and numero = 205}\\
    \textcolor{cyan}{\hspace{2em}commit work;\\
    end transaction}


    \vario{Proprietà delle transazioni}{Una transazione ha quattro proprietà:
    \begin{itemize}
    \item Atomicità
    \item Consistenza
    \item Isolamento
    \item Persistenza
    \end{itemize}
    }

    \subsection{Atomicità}
    Una transazione è una unità di esecuzione \textbf{indivisibile}. O viene eseguita completamente o non viene eseguita affatto.\par
    Implicazioni:
    \begin{itemize}
      \item Se una transazione viene \textbf{interrotta prima} del commit, il lavoro fin qui eseguito dalla transazione deve essere disfatto ripristinando la situazione in cui si trovava la base di dati prima dell’inizio della transazione. 
      \item Se una transazione viene \textbf{interrotta dopo} l’esecuzione del commit (commit eseguito con successo), il sistema deve assicurare che la transazione abbia effetto sulla base di dati.

    \end{itemize}
    
    \subsubsection{Consistenza}
  L’esecuzione di una transazione non deve violare i \textbf{vincoli di integrità}.\par
  Implicazioni:
    \begin{description}
      \item[Verifica immediata]:
      \begin{itemize}
        \item Fatta nel corso della transazione
        \item Viene abortita solo l’ultima operazione e il sistema restituisce all’applicazione una segnalazione d’errore
        \item L’applicazione può quindi reagire alla violazione.
      \end{itemize}
      \item[Verifica differita]: 
      \begin{itemize}
        \item Al commit se un vincolo di integrità viene violato la transazione viene abortita senza possibilità da parte dell’applicazione di reagire alla violazione
      \end{itemize}
    \end{description}
    

    

    \subsubsection{Isolamento}
    L’esecuzione di una transazione deve essere \textbf{indipendente} dalla contemporanea esecuzione di altre transazioni.

    Implicazioni:
    \begin{itemize}
      \item Il rollback di una transazione non deve creare rollback a catena di altre transazioni che si trovano in esecuzione contemporaneamente.
      \item Il sistema deve regolare l’esecuzione concorrente con meccanismi di controllo dell’accesso alle risorse
    \end{itemize}


    \subsubsection{Persistenza}
    L’effetto di una transazione che ha eseguito il commit non deve andare perso.
    Implicazioni:
    \begin{itemize}
      \item Il sistema deve essere in grado, in caso di guasto, di garantire gli effetti delle transazioni che al momento del guasto avevano già eseguito un commit.
    \end{itemize}

    \section{Istruzione SELECT}

    \begin{lstlisting}
    SELECT [ DISTINCT ]
    [ * | expression [[ AS ] output_name ] [, ...] ]
    [ FROM from_item [, ...] ]
    [ WHERE condition ]
    [ GROUP BY grouping_element [, ...] ]
    [ HAVING condition [, ...] ]
    [ { UNION | INTERSECT | EXCEPT } [ DISTINCT ] other_select ]
    [ ORDER BY expression [ ASC | DESC | USING operator ]]
    \end{lstlisting}

    \subsection{Struttura}

    \begin{description}
      \item[expression:] è un’espressione che determina un attributo.
      \item[from\_item:] è un’espressione che determina una sorgente per gli attributi.
      \item[condition:]è un’espressione booleana per selezionare i valori degli attributi.
      \item[grouping\_element:] è un’espressione per poter eseguire operazioni su più valori di un attributo e considerare il risultato.
    \end{description}

    \begin{flushleft}
        \textcolor{cyan}{SELECT} attributo\\
        \textcolor{cyan}{FROM} tabella\\
        {[} \textcolor{cyan}{WHERE} Condizione {]}\\
    \end{flushleft}

    \begin{itemize}
      \item SELECT, FROM e WHERE sono chiamate clausole
      \item Estrae dal prodotto cartesiano delle tabelle elencate nella clausola \textcolor{cyan}{FROM}, le righe che soddisfano la condizione specificata nella clausola WHERE.
      \begin{itemize}
        \item Il risultato dell‘interrogazione è una tabella con una riga per ogni riga prodotta dalla clausola \textcolor{cyan}{FROM} ed eventualmente filtrata dalla clausola \textcolor{cyan}{WHERE} (se esiste), le cui colonne dipendono dalla lista degli attributi nella clausola SELECT.
      \end{itemize}
    \end{itemize}

    \vspace{2em}

    \begin{lstlisting}
    SELECT [ DISTINCT ]
    [ * | expression [[ AS ] output_name ] [, ...] ]
    \end{lstlisting}

    \begin{itemize}
      \item \textcolor{purple}{*} è un’abbreviazione per indicare tutti gli attributi delle tabelle.
      \item \textcolor{cyan}{expression} è un’espressione che coinvolge gli attributi della tabella (può anche essere semplicemente il nome di un attributo).
      \item \textcolor{purple}{output\_name} è il nome assegnato all’attributo che conterrà il risultato della valutazione dell’espressione expression nella relazione risultato.
      \item \textcolor{purple}{DISTINCT}: se presente richiede l’eliminazione delle tuple duplicate.

    \end{itemize}

    \subsection{Operatore LIKE}
    \begin{itemize}
      \item Per il confronto tra stringhe, SQL mette a disposizione anche l’operatore \textcolor{purple}{LIKE} che consente verificare se una stringa rispetta un certo “pattern” (oppure \textcolor{purple}{ILIKE} per case insentive).
      \item Due caratteri speciali per il confronto “\_” e “\%”:
      \begin{itemize}
        \item “\_”: confronto con un carattere arbitrario
        \item “\%”: confronto con una stringa di lunghezza arbitraria (anche 0) di caratteri arbitrari
      \end{itemize}
      \item ‘\_o\%i’ = stringa che ha una o in seconda posizione, seguita da una sequenza qualsiasi di caratteri (anche vuota) e che termina con una i.
    \end{itemize}

    \subsection{Operatore SIMILAR TO}

      \item  L’operatore \textcolor{purple}{SIMILAR TO} è un \textcolor{purple}{LIKE} più espressivo che accetta, come pattern, un sottoinsieme delle espressioni regolari POSIX (versione SQL).
      \item Esempi di componenti di espressioni regolari:
      \begin{itemize}
        \item \_ = 1 carattere qualsiasi.
        \item \% = 0 o più caratteri qualsiasi.
        \item * = ripetizione del precedente match 0 o più volte.
        \item + = ripetizione del precedente match UNA o più volte.
        \item {n,m} = ripetizione del precedente match almeno n e non più di m volte.
        \item [...] = ... è un elenco di caratteri ammissibili
      \end{itemize}

    \subsection{Operatore IS NULL}
    \begin{itemize}
      \item Per verificare se un attributo ha un valore nullo o meno si utilizza il predicato \textcolor{purple}{IS NULL}.
      \item \textcolor{purple}{Attributo IS NULL}: ritorna true se l’attributo ha valore nullo.
      \item \textcolor{purple}{IS NOT NULL}: è la negazione, ritorna true se l’attributo ha valore diverso da nullo.
    \end{itemize}

     \begin{lstlisting}
    WHERE attributo IS [NOT] IN NULL;
    \end{lstlisting}

     SQL-2 implementa la logica a 3 valori:
    \begin{itemize}
      \item Un predicato semplice restituisce il valore NULL quando uno qualsiasi dei termini del predicato ha valore nullo.
      \item Il predicato IS NULL è una eccezione, restituendo sempre il valore vero oppure false, mai il valore NULL.
    \end{itemize}

L’istruzione SELECT, con una sola relazione nella clausola FROM permette di realizzare: \textcolor{black!40!green}{selezioni}, \textcolor{black!40!green}{proiezioni}, \textcolor{black!40!green}{ridenominazioni}.\\

    \subsection{Interrogazioni con JOIN}
In SQL il join si può definire in due modi diversi
\begin{itemize}
  \item Il primo non richiede l’uso di nuova sintassi, e specifica il legame tra le tabelle coinvolte a livello della clausola WHERE.
  \item Il secondo usa una sintassi riservata a livello di clausola FROM (useremo questo). 
\end{itemize}

 La clausola \textcolor{purple}{FROM} ammette come argomento:
  \begin{center}
    table\_name [[\textcolor{purple}{AS}] name]] [, ...] \\
  \end{center}
 Si è visto che se sono presenti due o più nomi di tabelle, si esegue il prodotto cartesiano tra tutte le tabelle e lo schema del risultato
può contenere tutti gli attributi del prodotto cartesiano. \\
Il prodotto cartesiano di due o più tabelle è un \textcolor{cyan}{CROSS JOIN}. \\
A partire da SQL-2, esistono altri tipi di JOIN (join\_type): \textcolor{cyan}{INNER JOIN}, \textcolor{cyan}{LEFT [OUTER] JOIN}, \textcolor{cyan}{RIGHT [OUTER] JOIN} e \textcolor{cyan}{FULL [OUTER] JOIN}.

  \qquad

\textbf{Sintassi generale}
\begin{lstlisting}
    table_name [NATURAL] join_type table_name [ ON join_condition [, ...]].
    \end{lstlisting}

dove \textcolor{cyan}{join\_condition} è un’espressione booleana che seleziona le tuple del join da aggiungere al risultato. Le tuple selezionate possono essere poi filtrate con la condizione della clausola \textcolor{purple}{WHERE}

\subsubsection{ SUM, MAX, MIN, AVG}


  \begin{flushleft}
<\textcolor{cyan}{SUM} | \textcolor{cyan}{MAX} | \textcolor{cyan}{MIN} | \textcolor{cyan}{AVG} > ( [\textcolor{cyan}{DISTINC} | \textcolor{cyan}{ALL}] AttrExpr )    \end{flushleft}


  \begin{itemize}
    \item SUM: restituisce la somma dei valori posseduti dall’espressione
    \item MAX/MIN: restituisce il massimo/minimo dei valori posseduti dall’espressione
    \item AVG: restiusce la media dei valori
  \end{itemize}

Altre caratteristiche sono:
  \begin{itemize}
    \item SUM e AVG ammettono come argomento solo espressioni che rappresentano valori numerici o intervalli di tempo
    \item MAX e MIN ammettono come argomento solo espressioni su cui è definito un ordinamento (quindi anche stringhe)
    \item DISTINCT elimina i duplicati
    \item ALL trascura solo i valori nulli
  \end{itemize}

\subsection{Interrogazioni nidificate}


Una interrogazione nidificata è una interrogazione che contiene al suo interno altre interrogazioni.\par
La nidificazione può avvenire sia nella clausola \textcolor{cyan}{SELECT}, che nella
clausola \textcolor{cyan}{FROM}, che nella clausola \textcolor{cyan}{WHERE}.
Il caso tipico è la nidificazione nella clausola WHERE, dove un
predicato contiene un confronto tra un valore (espressione su una
singola riga) ed il risultato di un’interrogazione SQL.

\subsubsection{Interrogazioni nidificate: WHERE}

Se in un predicato si confronta un attributo o una espressione che
produce un valore singolo con il risultato di un’interrogazione, sorge il problema di disomogeneità dei termini del confronto. \par
I normali operatori di confronto (=, <>, <, >, <=, =>) possono essere estesi con le parole chiave \textcolor{cyan}{ANY} e \textcolor{cyan}{ALL}.

\begin{itemize}
  \item \textcolor{cyan}{ANY}: la riga soddisfa la condizione se risulta vero il confronto (con l’operatore specificato) tra il valore dell’attributo (o espressione) e \textcolor{cyan}{ALMENO} uno degli elementi restituiti dall’interrogazione. 
  \item \textcolor{cyan}{ALL}: la riga soddisfa la condizione se risulta vero il confronto (con l’operatore specificato) tra il valore dell’attributo (o espression) e \textcolor{cyan}{TUTTI} gli elementi restituiti dall’interrogazione.
\end{itemize}
